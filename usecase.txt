Introduction, Scope & Process

1	Introduction:

GSRT Mailbox Auto-Response Automation is designed to streamline the handling of incoming emails by intelligently detecting and processing relevant messages. Upon identifying such emails, the system evaluates the content and associated ticket information to generate a context-aware, logical response. The BOT then sends an automated reply, guiding the user with appropriate instructions or actions tailored to the specific scenario

2	Process Description:

Incoming emails to the GSRT mailbox are scanned for ServiceNow Incident (INC)* or Service Request (ADH)* numbers in the subject or body. The system then verifies the corresponding ticket’s status in ServiceNow.
•	If the ticket is missing or marked as inactive (e.g., Resolved, Closed, or Cancelled), an automated and logical response is triggered to guide the user accordingly.
•	If the ticket status is New, Work in Progress (WIP), or On Hold, the system generates a context-aware response based on the predefined logic outlined in the automation design document.

3	Scope:

This automation applies to all emails received in the GSRT mailbox that are not from auto-notification sources. It covers identification of valid case numbers, checking their status in ServiceNow, and sending appropriate responses or escalating as per predefined categories.
























4	Process Flow Diagram


 





4.1	Process Flow Explanation

•	Read the GSRT Mailbox
The process reads all unread emails from the configured GSRT mailbox using MS Outlook VBO.  
•	Check if the Email is from a Bot/Auto-notification
•	If the sender’s email matches predefined bot or auto-notification patterns, or contains keywords (e.g. FSPROD,UNX,FSPRD,Appsrv,WebsrvService).  → No Response / No Action is taken.
•	If not, the process continues.

•	Check for Valid Incident and Ad-hoc Number
•	The process passes the email’s subject and body to extract a valid Incident Number and Ad-hoc Number.
•	If either number is missing, → Reply to user with: "No Ticket Found – Please Create One", then End.  
Note: Service Now URL to create the ticket will be shared with the User within the respond email.
•	Check the Incident State (via ServiceNow API)
The incident state is checked using a GET request to ServiceNow with the extracted Incident Number. 
State-Based Actions:
•	If Incident is Resolved / Cancelled / Closed: → Reply to user: "Ticket is already resolved. Please raise a new request ." → End process. 
Note: Service Now URL to create the ticket will be shared with the User within the respond email.
•	If Incident is New / In Progress / On Hold: → Reply to user: “We are reviewing your query” And notify the support team with ticket details to take further action. → End process   .

2. Data Security: 
All personal and sensitive user data must be securely handled and transferred. Ensure that data is encrypted during transmission, especially when interacting with external systems like ServiceNow.

3. Authentication and Authorization: 
•	Blue Prism use secure authentication mechanisms (e.g., OAuth tokens, ServiceNow API keys) when accessing ServiceNow and the internal user profile database.
•	Only authorized Blue Prism processes should have permission to modify or access sensitive data.

Design Considerations

1.	Design Principles

•	Automation-First Approach: Designed to minimize human intervention by auto-responding to emails based on ticket presence and status.
•	Rule-Based Logic: Uses keyword and pattern matching for classification (e.g., bot email, ticket number).
•	Business Continuity: Ensures critical user queries are acknowledged, and relevant teams are notified.
•	Scalability: Easily extendable for additional conditions, senders, or response types.
